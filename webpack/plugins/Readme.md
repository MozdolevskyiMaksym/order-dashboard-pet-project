# Webpack plugins

Цей розділ описує кастомні плагіни, які я використовую у проєкті. Кожен плагін має власний підрозділ з описом призначення, етапу роботи та формату вихідних даних.

---

## BuildMetaPlugin

**Коротко:** кастомний Webpack‑плагін, який під час production‑збірки генерує файл `build-meta.json` з метаданими білда.

### Навіщо

Плагін фіксує ключову інформацію про збірку, щоб:

- відслідковувати релізи та відповідність артефактів;
- спрощувати CI‑перевірки;
- швидше дебажити проблеми після деплою.

### Як працює

Під час збірки Webpack плагін:

- підʼєднується до lifecycle через `compiler` та `compilation`;
- працює на фінальній стадії `processAssets`;
- формує JSON‑файл з метаданими білда.

### Вміст `build-meta.json`

- режим збірки (development / production);
- час створення білда;
- версія Node.js;
- список згенерованих артефактів (JS, CSS, assets, static).

---

### Навіщо потрібен `build-meta.json`

Згенерований файл може використовуватись для:

- трасування релізів (яка саме збірка зараз у продакшені);
- перевірок у CI (наприклад, що збірка була саме production);
- аналізу проблем з кешуванням або чанками;
- аудитів і розборів інцидентів;
- автоматизації release notes або логування.

> Трасування релізів — це можливість точно знати, яка збірка, з якими артефактами і коли була задеплоєна в продакшені.

Таким чином, плагін додає інфраструктурну цінність, а не просто створює допоміжний файл.

---

### Результат

Плагін:

- демонструє розуміння Webpack lifecycle;
- використовує правильні API для роботи з assets;
- розширює процес збірки без порушення стандартного пайплайна;
- може бути повторно використаний або розширений у реальних проєктах.

## AnalyzerPlugin

**Коротко:** обгортка над `webpack-bundle-analyzer`, яка додає зручне керування режимами через конфіг або змінні середовища.

### Навіщо

`webpack-bundle-analyzer` допомагає відповісти на питання:

- що саме потрапило в бандл;
- чому бандл великий;
- які пакети “жруть” розмір;
- які чанки створились (code splitting).

Інструмент аналізує результат webpack‑збірки (stats) і робить:

- HTML‑звіт;
- веб‑інтерфейс;
- JSON/stats‑файл для інших інструментів.

### Ідея плагіна (простими словами)

Стандартний плагін зручний для разового запуску, але в реальному проєкті або постійно вмикається, або змушує правити конфіг. Я зробив власний `AnalyzerPlugin`, який успадковує базовий плагін і додає керування режимами.

Ідея проста:

- в одному місці описати, як саме може працювати аналізатор;
- керувати цим через конфіг або змінні середовища, а не через правки коду.

### Режими

- **off** — аналізатор не підключається;
- **static** — генерується HTML‑звіт з візуальним аналізом бандла;
- **server** — запускається локальний інтерфейс аналізатора;
- **stats/json** — генерується JSON‑файл зі структурою бандла для CI або інших інструментів.

За замовчуванням у продакшені аналізатор вимкнений, щоб не сповільнювати збірку. Коли аналіз потрібен, достатньо передати змінну середовища — і плагін автоматично працює у потрібному режимі.

### Чому це зручно

- логіка аналізу винесена в окремий модуль;
- конфігурація Webpack залишається чистою;
- поведінка збірки легко керується;
- один і той самий код підходить і для локальної розробки, і для CI.

Цей підхід дозволяє використовувати готовий інструмент, але адаптувати його під реальні потреби проєкту без переписування чужого коду.
