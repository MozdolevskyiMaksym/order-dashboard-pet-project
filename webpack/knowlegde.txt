1) Як працює конфігурація бандлера?

Webpack будує граф залежностей починаючи з entry. Далі він проходить по всіх імпортах (import ...) і застосовує:
	•	loaders для трансформації конкретних типів файлів (TS, CSS, тощо)
	•	plugins для керування процесом збірки на рівні всього бандла (генерація HTML, оптимізації, інʼєкція змінних)

Результат збірки потрапляє в output (dist/).

2) Як керувати залежностями коду (dependency relationship)?

Залежності задаються через модулі (import/export). Webpack автоматично визначає порядок підключення на основі графа залежностей.
Наприклад, якщо App.tsx імпортує styles.css і logo.png, то вони стають частиною графа залежностей і збираються/підключаються коректно.

3) Як завантажуються assets у правильному порядку (CSS/Images)?
	•	CSS підключається через import "./styles.css" — Webpack спочатку обробляє CSS loader’ами, а потім:
	•	у dev інжектить в <style> (через style-loader)
	•	у prod виносить в окремий .css файл (через MiniCssExtractPlugin)
	•	Зображення підключаються через import logoUrl from "./logo.png" і обробляються через asset/resource, після чого Webpack генерує файл у dist/assets і повертає URL.

4) Як створити кілька конфігурацій для різних середовищ?

Використано підхід:
	•	webpack.common.js — спільні налаштування (entry/output/TS/assets/plugins)
	•	webpack.dev.js — dev налаштування (dev server, HMR, devtool, CSS через style-loader)
	•	webpack.prod.js — prod налаштування (CSS у файл, optimization splitChunks/runtimeChunk)

Конфіги обʼєднуються через webpack-merge.

5) Переваги використання plugins

Plugins дозволяють робити речі, які складно або неможливо зробити лише loader’ами:
	•	генерувати HTML та автоматично підключати бандли
	•	інʼєктити константи/змінні в код під час збірки
	•	витягувати CSS в окремий файл
	•	робити глобальні оптимізації (splitting, minification, аналіз бандла тощо)

6) Різниця між loaders і plugins
	•	Loaders працюють з окремими файлами і трансформують їх (наприклад, TS → JS, CSS → JS/CSS, обробка asset’ів)
	•	Plugins працюють на рівні процесу збірки в цілому (генерація HTML, оптимізації, інʼєкція env, екстракція CSS тощо)


### Чому корисно аналізувати бандл?
Аналіз бандла дозволяє зрозуміти:
- які залежності займають найбільше місця
- як працює code splitting
- які частини коду можна оптимізувати або винести окремо

У цьому проєкті використано `webpack-bundle-analyzer` для візуального аналізу результату production-збірки.



## Чи є мінімізація, tree-shaking і прод-оптимізації в dev режимі?

Ні. Або майже ні.
Тому що dev і prod мають різні цілі.

Dev режим - головне:
  •	швидкість
	•	зрозумілі помилки
	•	зручний дебаг

У dev:
	•	код не мінімізується
	•	змінні, функції, назви не стискаються
	•	tree-shaking або вимкнений, або працює дуже обмежено
	•	багато інформації для дебагу
	•	швидкі source maps

Важливо бачити код, а не мати найменший файл

Production режим — головне:
	•	розмір файлів
	•	швидкість завантаження
	•	стабільність

У prod:
	•	код мінімізується
	•	зайві частини коду викидаються (tree-shaking)
	•	оптимізується порядок і структура чанків
	•	браузер отримує менше байтів


Що таке source map і навіщо він потрібен

Source map - це спеціальний файл, який містить відповідність між зібраним (мінімізованим) JavaScript-кодом і оригінальним вихідним кодом (TypeScript, TSX тощо).
Він дозволяє браузеру та інструментам розробника показувати реальні файли та рядки вихідного коду замість складного зжатого JavaScript.

Під час production-збірки код оптимізується: мінімізується, перейменовуються змінні, кілька файлів зʼєднуються в один. Без source map у випадку помилки браузер може показати лише номер символу в одному великому JS-файлі, що робить дебаг майже неможливим. Source map вирішує цю проблему, дозволяючи бачити помилки у контексті оригінального коду.


Як source map використовується на практиці

Коли source map доступний:
	•	у вкладці Sources DevTools відображаються .ts / .tsx файли
	•	помилки та stack trace вказують на реальні файли і рядки
	•	системи логування помилок (наприклад, Sentry) можуть показувати зрозумілий стек викликів

Source map не використовується користувачем і не впливає на роботу застосунку. Він потрібен виключно для дебагу та аналізу помилок

Source map у development режимі

У development-режимі source maps:
	•	увімкнені завжди
	•	максимально детальні
	•	швидкі для перебудови
	•	відкрито доступні браузеру

Це необхідно для зручної локальної розробки, швидкого дебагу та роботи з вихідним кодом.

⸻

Source map у production режимі

У production source maps:
	•	можуть містити повний вихідний код
	•	можуть розкривати внутрішню логіку застосунку
	•	не повинні бути відкрито доступні кінцевому користувачу

Тому в продакшені зазвичай використовують:
	•	hidden-source-map — source map створюється, але браузер не підключає його автоматично
	•	або source maps взагалі не публікуються

У такому випадку source maps використовуються лише для внутрішнього аналізу помилок або інтеграції з системами моніторингу.
